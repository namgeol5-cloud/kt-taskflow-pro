# 00-overview — TaskFlow Pro 문서 개요

## 프로젝트 한 줄 정의

**TaskFlow Pro** — 팀 업무 관리 풀스택 웹앱

---

## 문서 구성 매핑표

| 파일명 | 역할 | 알 수 있는 것 |
|---|---|---|
| `00-overview.md` | 문서 전체 안내 | 파일 구성 의도, 읽는 순서, 분리 원칙 |
| `01-product.md` | 제품 정의 | 타겟 사용자, 핵심 기능, 비즈니스 목표 |
| `02-specs.md` | 기술 명세 | 기술 스택, 버전, 시스템 아키텍처, API 설계 |
| `03-design.md` | UI/UX 설계 | 화면 구성, 컴포넌트 구조, 디자인 원칙 |
| `04-tasks.md` | 작업 계획 | 개발 로드맵, 마일스톤, 우선순위 |
| `05-conventions.md` | 코드 규칙 | 네이밍 규칙, 폴더 구조, 커밋 컨벤션, 환경변수 관리 |

---

## 읽는 순서

```
00-overview  →  01-product  →  02-specs  →  03-design  →  04-tasks  →  05-conventions
   (지도)          (목적)         (구조)        (화면)        (계획)         (규칙)
```

새 작업을 시작하기 전 **00 → 05 순서대로** 읽는다. 중간을 건너뛰면 뒷 문서의 맥락이 끊긴다.

---

## 왜 이렇게 나누는가 — 관심사 분리 원칙

코드에서 레이어를 분리하듯, 문서도 **하나의 파일이 하나의 관심사만 담는다**.

| 원칙 | 적용 |
|---|---|
| 변경 이유가 다르면 파일도 다르다 | 기능 요구사항이 바뀌면 `01-product`만 수정, 기술 스택이 바뀌면 `02-specs`만 수정 |
| 탐색 비용을 줄인다 | "UI 규칙이 어디 있지?" → `03-design`, "배포 환경변수?" → `05-conventions` |
| 리뷰 범위를 좁힌다 | PR에서 변경된 문서 파일만 보면 무엇이 결정됐는지 즉시 파악 가능 |
| 신규 합류자의 온보딩을 돕는다 | 00부터 읽으면 전체 맥락을 순서대로 습득할 수 있다 |

한 파일에 모든 내용을 몰아 넣으면 수정 충돌이 잦고, 어디를 봐야 할지 몰라 문서가 결국 방치된다.
관심사를 분리하면 **문서가 코드처럼 유지보수된다**.
